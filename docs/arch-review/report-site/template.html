{{OVERVIEW}}

How this review was produced

This was not a single-pass read. It is a multi-fleet review run as a deterministic Claude Code Workflow over the Scout codebase, designed after two earlier single-agent runs overlapped only ~60% and each confidently misdescribed code it had read. The design goal: a real miss should require three independent failures.

The pipeline · five phases
Round-1 review fleet

Three overlapping partitions of the codebase — by module, by feature, and by failure class — plus seam, journey and history mandates. ~25–30 reviewers in round 1, then targeted gap rounds until dry.

Key & glossary

Every finding is scored on two independent axes — status (how alive the problem is) and impact (what's at stake) — then carries an adversarial-verification verdict, a complexity verdict, and a replication count. Here is what each term means.

Status

How alive is it — does it bite now, later, or just rot?
BROKEN-NOW
Reachable and failing today. A live route, UI control, or agent tool hits this path right now and it produces wrong behavior. No special conditions required.
LATENT
A real defect that hasn't fired yet. It needs a specific trigger to surface — a race, a multi-tenant workspace, a TTL expiry, an error path, an upstream change — but when that condition occurs, it bites.
DEBT
Not actively wrong, but structural rot. Dead code, the same truth implemented N different ways, missing tests, drift that hasn't broken anything yet. It raises the cost and risk of every future change.
COSMETIC
Minor and low-impact. Naming residue, stale docstrings, small UI imperfections. Worth recording, not worth an incident.

Impact

If it fires, what is at stake?
data-loss
Destroys or corrupts data — e.g. dropping a database schema that still holds rows.
security
Authentication, authorization, tenant isolation, or information disclosure.
correctness
Produces wrong results or wrong behavior, without destroying data.
cost-perf
Costs money or latency — LLM bill, redundant requests, slow paths, unbounded growth.
velocity
Slows development down — confusing code, fragile contracts, missing guardrails.

Verdict

Adversarial verification — every claim was independently re-traced by a verifier whose job was to refute it.
CONFIRMED
The verifier re-traced the path and the claim holds as stated.
PARTIAL
True, but narrower than first claimed. The verifier corrected the severity or reach; the narrowed claim is what is shown here.
REFUTED
The claim was overturned on trace. The two refuted findings are retained in Appendix B for the record, not in the body.

Complexity

Is the difficulty inherent to the domain, or self-inflicted?
accidental
Avoidable. Drift, rename residue, a fix applied only where it bit, N parallel implementations of one truth. The dominant verdict — most issues are fixable rot, not hard problems.
mixed
Some inherent difficulty, some avoidable.
essential
Inherent domain difficulty — genuinely hard regardless of design choices.

Replication

A free confidence signal.
rK
How many of the independent reviewers found this same issue on their own. Higher means more confident it is real: 23 findings were found by ≥5 reviewers, and all 23 survived verification. Security and data-loss findings additionally got two independent verifiers.
NN#i
Finding id. NN is the source batch file (findings/batch-NN.json) and i is the array index — e.g. 00#0 is the refresh data-loss finding. These ids appear as clickable links throughout the narrative.
